# Quality and performance benchmarks vs R-INLA

Plan for showing users *what they're getting* relative to R-INLA. Two
deliverables on different timelines:

- **Quality benchmark** — ships in v0.1.0 as a docs page. Half-day of
  work. Reuses existing oracle fixtures.
- **Performance benchmark** — deferred to a v0.1.x patch (target
  v0.1.1, ~2026-05). Needs CI infrastructure for R-INLA.

The split is per the recommendation in
[plans/release-v0.1.md](release-v0.1.md): credibility for v0.1.0
demands quality numbers in user-facing docs, but a half-baked
performance table hurts more than no performance table.

---

## Part 1 — Quality benchmark (v0.1.0)

### Goal

A single docs page (`docs/src/benchmarks/quality.md`) that, for three
canonical R-INLA examples, shows side-by-side posterior summaries
from Julia INLA and R-INLA, plus the tolerance status from the oracle
test suite.

### Datasets

The three R-INLA examples already shipped as oracle fixtures:

| Example | Fixture | Test file |
|---|---|---|
| Scotland lip cancer (BYM2) | `packages/LatentGaussianModels.jl/test/oracle/fixtures/scotland_bym2.jld2` | `test_scotland_bym2.jl` |
| Pennsylvania lung cancer (BYM2) | `packages/LatentGaussianModels.jl/test/oracle/fixtures/pennsylvania_bym2.jld2` | `test_pennsylvania_bym2.jl` |
| Meuse zinc (Gaussian + SPDE) | `packages/INLASPDE.jl/test/oracle/fixtures/meuse_spde.jld2` | `test_meuse_spde.jl` |

These cover the three structural axes of the v0.1 ecosystem: areal
disease mapping with a single connected component (Pennsylvania),
areal with multiple components (Scotland), and geostatistics
(Meuse). One each — no synthetic fixtures, no `@test_broken`
items as the headline.

### Page structure

Per dataset, three blocks:

1. **Model recap** — one paragraph, link to the corresponding
   vignette (which already exists for all three).
2. **Posterior summary table** — fixed-effect means/SDs and
   hyperparameter means on the user scale (e.g. `(ρ, σ)` for SPDE,
   `τ_b, φ` for BYM2), Julia vs R-INLA in adjacent columns, with the
   relative error in a third column.
3. **Tolerance status** — the assertion bands from
   `plans/testing-strategy.md` (1% on fixed-effect means, 5% on
   hyperparameters, looser bands explicitly noted), plus a one-line
   summary of any `@test_broken` divergences for that dataset (links
   into `plans/replan-2026-04.md` "Open follow-ups" §).

A footer **summary table** at the top of the page rolls up: dataset,
"all assertions pass" / "N broken", link to the per-dataset detail.

### Implementation

`docs/src/benchmarks/quality.md` is a Documenter `@example` page that
loads each fixture, runs `inla()`, and renders the comparison table
inline. The fixtures are tiny (kB-scale) and the fits are sub-minute
each, so the docs build cost is acceptable.

A small helper
`docs/src/benchmarks/_compare.jl` provides
`quality_table(julia_summary, r_summary; bands)` that emits a
GitHub-flavoured markdown table — keeps the per-dataset
`@example` blocks short and uniform.

### Acceptance

- [ ] `docs/src/benchmarks/quality.md` renders under `docs/build/`.
- [ ] Page is linked from the Documenter sidebar (under a new
      "Benchmarks" section, between "Vignettes" and "Packages").
- [ ] Each dataset's comparison agrees with the assertions in its
      oracle test (no silent contradiction between docs and CI).
- [ ] `plans/release-v0.1.md` pre-flight checklist gains a tickbox
      for "quality benchmark page is current".

### Effort

Half-day. Existing oracle fixture loaders, existing `inla()` calls,
new code is mostly the comparison-table helper and three short
`@example` blocks.

### Non-goals (for v0.1.0)

- Wall-clock numbers. Deferred to Part 2.
- Synthetic-fixture tables. The oracle tests already cover these;
  not load-bearing for user trust.
- Comparison across solvers (CHOLMOD vs Pardiso, etc.). Pardiso is
  cut from v0.1 per ADR-015.

## Part 2 — Performance benchmark (v0.1.x patch, target v0.1.1)

### Why deferred

A credible perf comparison needs: (a) R + R-INLA installed in a
reproducible environment, (b) statistical methodology that respects
multi-run noise (BenchmarkTools median + IQR, not single timings),
(c) explicit hardware spec, (d) version pins on both sides. Getting
this right is **at least a week of CI plumbing** and an ongoing
maintenance cost. v0.1.0 should not block on it.

### Goal

A `benchmarks/` directory at the repo root with a runnable harness
that reports, for the same three R-INLA examples used in Part 1:

- Wall-clock to fit (`inla()` end-to-end, R-INLA `inla()` end-to-end).
- Memory peak (Julia: `BenchmarkTools.@benchmark` allocs; R: `Rprof`
  memory profiling).
- Numerical agreement summary (link to Part 1's quality numbers).

Reported as a single GitHub-rendered markdown table, regenerated by a
documented `julia --project=benchmarks benchmarks/run.jl` command.

### Architecture

```
benchmarks/
├── Project.toml             # BenchmarkTools, RCall, the four packages
├── run.jl                   # entry point: runs all benchmarks, writes results.md
├── harness.jl               # Julia-side timing + memory measurement
├── r_inla.R                 # R-INLA fits for the same three datasets
├── compare.jl               # parses both sides, emits the table
└── results/
    ├── 2026-05-15_x86_64.md  # one results file per (date, hardware)
    └── ...
```

Results are versioned in-repo so the user / reader can see how the
gap evolves over releases.

### Methodology constraints

- **Same input.** Both sides start from the same oracle fixture
  (`scotland_bym2.jld2` etc.). No re-derivation of `W`, `E`, etc.
- **Warm-up.** Julia: `BenchmarkTools` defaults (5 s warm-up,
  median over `n` samples). R-INLA: discard first run.
- **Pinned versions.** R-INLA version pinned in
  `benchmarks/r_inla.R`; Julia versions in
  `benchmarks/Manifest.toml`. Versions reported in every results
  file's header.
- **Hardware spec in every results file.** CPU model, core count,
  RAM, OS. The `results/<date>_<arch>.md` filename tracks this.
- **Single-threaded comparison by default.** Both sides forced to one
  BLAS thread to remove parallelism noise from the comparison.
  Multi-threaded runs are a separate table in the same file.

### CI integration

Optional GitHub Actions workflow (`.github/workflows/benchmarks.yml`)
that runs on a manual trigger only — *not* on every push. Manual
trigger writes a fresh `results/<date>_ci.md` and opens a PR. CI
hardware is noisy; the published numbers should come from a known
machine, not GitHub runners.

### Acceptance (for v0.1.1 cut)

- [ ] `benchmarks/run.jl` produces a results file for the three
      R-INLA examples on at least one documented hardware spec.
- [ ] Results file is linked from `docs/src/benchmarks/quality.md`
      (the v0.1.0 page) under a "Performance" §, replacing the
      "deferred to v0.1.x" placeholder.
- [ ] R-INLA version + Julia versions + hardware are visible in the
      results file header and explicitly cited from the docs page.
- [ ] No performance claims on the docs page that aren't backed by a
      results file.

### Effort

~1 week wall-clock:
- 2 days to build the harness and verify it on Scotland.
- 1 day to add Pennsylvania and Meuse.
- 1 day to write the comparison report and integrate with docs.
- 1 day buffer for CI plumbing and hardware-spec pinning.

### Non-goals

- Auto-published benchmark history with charts. Single results files
  per release are enough for v0.1.x.
- Comparison against NIMBLE / Stan-IAR. Tier-3 triangulation is a
  separate effort tracked in `plans/testing-strategy.md`.

## Sequencing relative to v0.1.0

1. **Now (before v0.1.0 tag):** ship Part 1 — `docs/src/benchmarks/
   quality.md`. ~half-day.
2. **v0.1.0 tagged.**
3. **v0.1.1 patch (target ~2026-05-15):** ship Part 2 —
   `benchmarks/` harness + first results file. Patch release because
   it's docs / CI, no API change.

table (no `@example`, just transcribed numbers from the fixtures).
The point is *user-visible quality numbers in v0.1.0*, not a perfect
auto-rendered page.
